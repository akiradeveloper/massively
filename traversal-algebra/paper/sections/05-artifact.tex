The proof project contains no admitted theorem and produces no executable
oracle.  Its responsibility ends at machine-checking the mathematical language
and abstract resource results.  Implementation validation is deliberately
independent: a small sequential Rust oracle owns raw CSR arrays and directly
evaluates ordered traversal, source/destination/edge reads, products, maps,
emission, and both reductions.  It shares no traversal or reduction code with
Massively.

A property-based Rust harness generates valid CSR graphs, frontiers, and value
columns, evaluates them with this CPU reference, and compares the results with
the public Massively GPU operations.  Small structural cases shrink at the
graph and frontier level; seeded multigraph, hub, bipartite, regular, and
parallel/self-loop families supply boundary coverage.  A separate deterministic
case contains 65,537 vertices and 393,222 edges so scale coverage does not
compromise counterexample shrinking.

Passing finitely many generated inputs demonstrates agreement between the two
implementations on those inputs; it does not prove that either Rust
implementation refines the Lean model.  The implementation-test trusted
boundary includes the CPU reference, Rust test harness, CubeCL, and the device
stack.  End-to-end verification of that stack is explicitly outside the
theorem's scope: the property tests provide engineering evidence, while
Theorem~\ref{thm:ta-bsp-equivalence} concerns the mathematical languages.  The
symbolic CubeCL resource equations remain universal Lean theorems rather than
runtime certificates transported through the test harness.
